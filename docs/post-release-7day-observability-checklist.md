# SUB2api 上线后 7 天观察指标清单

版本基线：`v0.1.92-rc.3`
适用环境：`staging` / `production`
用途：上线后连续 7 天执行稳定性与风险收敛观察，形成可审计证据。

## 执行原则
1. 每日固定时段（建议 10:00 与 18:00）巡检一次。
2. 指标异常必须记录“发现时间 / 影响范围 / 处置动作 / 恢复时间”。
3. 所有结论以可追溯数据为准，不接受口头通过。

## Day 0（发布当天）
1. 发布链路
- [ ] Release tag 与 Release 页面一致（tag、commit、发布时间一致）。
- [ ] Deploy Marker（staging / production）均为 `success`。
- [ ] 对应 deployment id 已记录到运维日志。

2. 入口可用性
- [ ] `/health` 连续 15 分钟探测成功率 `>= 99.9%`。
- [ ] 关键 API（鉴权、账号、网关）冒烟通过。

3. 配置基线
- [ ] 生产环境关键 secret/var 与基线一致：
  - `POSTGRES_PASSWORD` / `JWT_SECRET` / `TOTP_ENCRYPTION_KEY` 非空
  - `SECURITY_URL_ALLOWLIST_ENABLED=true`
  - `SECURITY_URL_ALLOWLIST_ALLOW_INSECURE_HTTP=false`
  - `SECURITY_URL_ALLOWLIST_ALLOW_PRIVATE_HOSTS=false`

## Day 1-3（稳定性观察）
1. 请求质量
- [ ] HTTP 5xx 错误率 `< 1%`
- [ ] P95 延迟不高于发布前基线 `+20%`
- [ ] P99 延迟未出现持续恶化（连续 30 分钟异常）

2. 资源与依赖
- [ ] CPU / 内存无持续爬升（无明显泄漏趋势）
- [ ] 数据库慢查询数量在可控阈值内（无爆发性增长）
- [ ] Redis/队列无明显积压
- [ ] 上游依赖（OpenAI/Anthropic/Gemini）失败率在可接受范围

3. 安全与治理
- [ ] required checks 正常执行并为合并门禁
- [ ] `upstream-sync` 定时任务持续成功
- [ ] 无未处理高危告警（安全扫描/依赖漏洞）

## Day 4-7（容量与回归观察）
1. 流量与成本
- [ ] 峰值流量下吞吐稳定，无雪崩
- [ ] 账单/成本曲线符合预期，无异常突增
- [ ] token 相关响应在 24h 维度内稳定（无长时间降级）

2. 数据一致性
- [ ] 核心业务数据（日汇总、统计报表）与原始日志一致
- [ ] 关键任务（定时任务、重试补偿）执行成功率达标

3. 变更控制
- [ ] 仅允许修复型变更进入主干
- [ ] 每次修复后都有冒烟回归记录

## 异常分级与响应
1. P0（阻断）
- 现象：服务不可用、核心链路中断、数据损坏风险
- 响应：立即回滚或降级，30 分钟内完成第一次状态通报

2. P1（高风险）
- 现象：错误率升高、明显性能退化、依赖持续不稳定
- 响应：当日完成缓解与复测，必要时冻结新变更

3. P2/P3（一般）
- 现象：局部功能缺陷、非关键性能抖动
- 响应：纳入迭代修复并跟踪关闭

- 异常与处置：
- 结论（继续观察 / 风险收敛 / 触发回滚预案）：

## 7 天收口标准
1. 连续 7 天无 P0 未闭合问题。
2. P1 全部关闭，或有明确风险接受单与负责截止时间。
3. 关键 SLO 达标（错误率、延迟、恢复时间）。
4. 形成完整证据包并归档到 `docs/uat-evidence/` 或运维审计目录。
